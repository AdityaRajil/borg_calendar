BORG Palm Conduits

Status: Beta

Conduits are available to sync the BORG appointment database to the Palm datebook and todo applications. These conduits replace the existing Palm Desktop conduits and cannot run side-by-side with them,  because Palm hot-sync does not allow 2 conduits to sync to the same database. In other words, if you install the BORG conduits, then Palm will no longer sync with Palm Desktop for datebook and todos.

These conduits are to be considered beta for 2 reasons:

  • They have only been tested on Palm OS 5.2 with datebook version 4.5. It is possible that newer versions of the Palm software will be problematic.
  • The BORG and Palm applications do not match in functionality and database structure.

I've been syncing my Palm-based Garmin iQue 3600 PDA since January 2005. I have well over 4000 appointments, and everything seems ok based on the way I use BORG and Palm.

Required Software

  • Windows
  • BORG 1.5
  • Palm OS 5 (only 5.2 tested)
  • Palm JSync Suite
  • Java Conduit Manager (www.javaconduitmanager.org) or alternative conduit installer

Linux is not supported.

Who should use this

These conduits are meant for BORG users who want to put their data on the Palm...as best as can be done...but who consider BORG to be the master database, and are willing to live with a few compromises on the Palm in order to use BORG. If you are a Palm Desktop user and just want a better GUI on the PC - don't use this. BORG is not a "drop-in" replacement for Palm Desktop. You should look for other apps that are written to provide an alternative GUI to Palm desktop.

If you are a BORG user who really likes BORG and you just want to be able to get your data onto the Palm any way you can - then these conduits are for you.

 

Warnings and Limitations

The functionality of BORG and Palm do not match - just like Outlook and Palm do not exactly match. BORG is NOT a GUI front end for the Palm datebook and todo apps. So, certain functionality in BORG will not work on the Palm and vice versa. For example, if you try to use a recurrence option in Palm that is not supported by BORG, the resulting BORG appointment will repeat incorrectly. The same is true if you use a BORG recurrence option that Palm does not support.

There are some BORG data items that have no corresponding fields in the Palm databases. For example - vacation, holiday, category, color - (my version of Palm datebook does not support these, but others might). This data will be stored on the Palm in the Notes field. Therefore, you cannot use the Notes field in the datebook and todo once you start syncing with BORG.

Installation Steps

First, the development version of these conduits and BORG 1.5 are intended for users who know how to back up their Palm data and restore it. If you start using the BORG conduits and wish to go back to Palm desktop, you must feel comforable that you know how to restore the original Palm. If you don't feel comfortable, then stop now. 

 

As of BORG 1.5.2, I recommend using the socket-based sync only. This is the only version that I plan to support from now on. The instructions below are for the socket-based sync. When using the socket-based sync, the Palm conduit will not access the BORG database directly, but will sync with BORG by communicating with a running instance of BORG. While the sync is in progress, the user will be locked out of BORG and a sync log will be shown in BORG to track the progress of the sync.

1.      Get BORG 1.5.2 or later and install. Choose to install the optional conduits.

2.      Get the java conduit manager (http://www.javaconduitmanager.org/downloads.htm). Get the larger download that includes JSync - unless you already have Jsync installed. Install jsync and java conduit manager.

3.      The BORG conduits require a borg folder under your Palm user folder. So create this folder.

4.      Check the new option in BORG to support Palm hot-sync (under miscellaneous).

5.      Enter a socket port under the miscellaneous options. Any unused port will do. I randomly chose 2929.

6.       RESTART BORG.

7.      Create a file called db.properties in the borg folder that you just created. In it put the following:

dburl=remote:xx
user=$default
port=2929 (or whatever number you chose for the port)

Shut down your hot-sync manager.

8.      Use the Java conduit manager to install the conduits, which can be found in a conduits folder where you installed the BORG software. Only install the conduits you want to use. The apptconduit replaces the palm datebook conduit. The addrconduit replaces the Palm addressbook conduit.

9.      Restart the hot-sync manager

10. Under the hot-sync manager custom menu – configure the To Do conduit to do nothing. The BORG apptconduit syncs to BOTH the datebook and todo list.

11. For the first sync, if you want all BORG records to overwrite the Palm, then go to the custom option of the hot-sync manager and change the hot-sync action for the BORG appointment conduit to the option that WIPES the Palm db. Do not make this the default action. This will do a one-time wipe that will copy your BORG data to the Palm and all pre-existing Palm appointments will be deleted!!!!!!

If you want to transfer existing Palm records to an empty BORG db, then just keep the normal sync action.

12. Press your hot-sync button to sync

13. Start BORG to check the results. Also check your Palm.

14. Continue to test and report problems on the BORG sourceforge bug tracker.

Using the socket-based sync as described above, BORG must be running during the sync.

If something has gone wrong, then you can check the hot-sync log (View Log menu option in Hot-Sync Manager).

You should be able to back out the BORG conduits using Java Conduit Manager.

Use the Archive Record Option

The datebook has the concept of archived records. The purpose of telling the Palm datebook to archive a record is to cause it to be saved to an archive file on the PC. Through trial and error, I have discovered that the Palm datebook (at least my version) does not seem to correctly send deleted appointments back to BORG unless you mark them for archiving. So always check the archive option when deleting datebook records on the Palm.

Alarms

In BORG 1.5, there is a new appointment option checkbox labeled "Alarm". This option does nothing at all in BORG. However, if you check this, then the appointment will have its alarm set in the Palm datebook when you sync to the Palm.

 

Background Information

What is a conduit

A conduit is a program that is invoked when you hot-sync a palm PDA. It is responsible for syncing 1 Palm database. You can see a list of the conduits if you invoke the "custom" menu option of the palm hot-sync manager.

How does a normal conduit work?

A normal conduit syncs 2 databases that HAVE THE SAME DB FIELDS. When the DB on the Palm and the DB on the PC have the same exact fields and keys, the conduit's job is simple. It does not matter if the data is stored on the PC using Palm .dat files, XML files, or even a relational database. The algorithm for matching records between the PC and Palm is simple, as is the mapping of data from one store to another. Usually, the applications that run off of the PC in this case are developed along with the Palm apps. For example, the Palm Desktop is developed to match (capability wise) with the Palm datebook, todolist, and address book. You also have third-party PC apps developed to match the existing Palm apps - for example, a new non-Palm calendar program developed on the PC that is developed from scratch to match the Palm datebook DB format, but to provide a better GUI than Palm Desktop. There are also linux apps that presumably run off of the Palm db formats and provide alternate desktops.

Are BORG conduits normal?

No, BORG has what I call asymmetric conduits. An asymmetric conduit must sync between 2 databases where the data DOES NOT MATCH. Here is why the BORG conduits are asymmetric:

  • The keys to the records in the BORG database have nothing in common with the keys in the Palm database.

 

  • BORG appointments and Palm datebook appointments contain different fields. For example, BORG appointments have a color - which some Palm datebooks might support, but not mine. Palm datebook records have repeat types that BORG does not. BORG supports holiday and vacation appointments....
  • The Palm address book contains different fields than the BORG address book.
  • My Palm is a GPS (Garmin 3600). The custom datebook and addressbook let you attach GPS data to an address or appointment. BORG certainly does not.
  • BORG todo items are appointments with all of the data that appointments have (color, vacation, time, etc...). Palm todo items are very simple and lack a place to put most of the BORG fields.
  • Even worse - BORG does not really have a separate Todo database like the palm. BORG really has 1 datebook/todo db, while the Palm has 2. So 2 palm DBs must sync to 1 BORG DB.
  • Palm categories are kept in a list outside of the databases - sort of a second database that needs to be synced, while BORG keeps category information in each record.

So why not make normal conduits for BORG? - that would entail either re-writing BORG to match the data structure and capabilities of the Palm desktop (not going to happen) or writing new apps that run on the Palm itself using the BORG db formats (not going to happen).

So, how to get the conduits to work? Well, I'll cover each in detail below, but in general, BORG is going to put its DB keys somewhere in the Palm records. In addition, any data that is in one DB and not in the other will have to be kept somewhere, and the conduit will have to sync very carefully as to not lose this data. Finally, the conduits will have code to convert between data items that are in both DBs, but that just don't match.

The Datebook Conduit

For datebook items (appointments), BORG will be the "master" in the sync relationship. You will be able to use BORG to create appointments any way you want - and they will be synced to the Palm. You can use the Palm to create/edit appointments - but not any way you want.

Why? Two reasons. First, the conduit will need to store the "asymmetric" BORG information somewhere on the Palm. This will be done in the datebook NOTES field. So you can't use the notes field for your own notes. The conduit will store a string in the notes field that will contain the BORG DB key, the appointment color, and flags to indicate Todo, Vacation, and holiday. The BORG key is needed on the Palm to match the record to the BORG db. If you edit a BORG record on the Palm, the conduit needs to know which BORG record needs to be updated. During such an update, the Palm record overwrites the BORG record (if the BORG record has not changed), so the conduit will take the other fields stored in the notes (color, vacation,...) and use these to complete the BORG record.

The other reason BORG is the master? BORG does not support all of the repeat options of the Palm. If you pick a repeat option on the Palm that does not match one in BORG, your appointment will not repeat at all on BORG.

These issues may be solved one day, but for now, you have to be aware of a few things when editing appointments on the Palm - don't edit the notes, and repeat carefully.

The Todo Conduit

The Palm Todo Conduit should be shut off. The BORG datebook (appt) conduit will sync to the todo database on the palm as well as the datebook database.

 

For todo items, changes made on the Palm to the Palm todo DB are first synced to BORG and then the Palm todo DB is wiped. Then the entire todo list from BORG is sent to the Palm. Since todo and appt records are the same thing in BORG, you actually wind up with todo appts being written to both the datebook and todo databases on the Palm.

So what can you do on Palm? Almost everything, actually. You can create new todos on the Palm and they will be written to BORG as new Todo appts. You can set todo categories on the Palm and these will become new BORG categories. You can edit existing todos on the Palm and the changes will be sent to BORG (only date, text, and category changes - priority is ignored). You cannot edit the Todo notes. The conduit will use the notes field to keep the BORG key on the Palm and also a date to help BORG know what is going on with repeating todos (which is a BORG feature, but not a Palm feature)

When you complete a todo on the Palm, the corresponding todo in BORG will be completed with the "no-delete" option.

The Address Conduit

For the address conduit the fields don't match up well. For address items, BORG and Palm are equals. The BORG addressbook is not good enough to favor it over the Palm addressbook. The only fields you must avoid on the address book are the custom ones. Custom 1 will contain the BORG key. Custom 2 will contain the BORG birthday. Custom 4 will contain the address notes. In cases where the fields do not match up, you should not lose any Palm data - i.e. BORG will not overwrite Palm fields it does not understand (I think). But the data might be missing on BORG or show up in the wrong place. The addressbook lets you use certain fields for multiple purposes - and the conduit does not handle this well.
 
If you are a Palm Desktop addressbook user and are happy with it, then don't use the BORG address conduit.
