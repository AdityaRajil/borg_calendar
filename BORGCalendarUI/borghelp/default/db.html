Databases

When BORG starts for the first time, it prompts for the location of the database to use to store appointments, tasks, and addresses.

At this time, the available choices are:

·        Local Files

·        RDBMS (MySQL)

Local Files

Almost all BORG users make use of the file-based databases provided as the default.

If the user selects local files from the database options screen and sets a valid local folder, BORG will create 3 files in that folder:

·        borg.jdb for storing calendar appointments

·        mrdb.jdb for tasks

·        addr.jdb for address/contact info.

These files are in a binary format. I would suggest backing up these files every so often if you use BORG, in case of disk failure (not because BORG is buggy :)

The database files are not simple data stores. These files are formatted to be highly resistant to corruption from disk space overruns and computer crashes. They are random access databases. Many calendar programs will overwrite the entire database file when there are any updates. This is especially true of iCal based programs that use local files. This is not only wasteful of computer resources, but is prone to catastrophic failure and loss of all data if something goes wrong during the overwrite. When a BORG jdb file is updated, file I/O is only done to the single record being updated. Even if the computer is shut off during the update, at most the 1 updated record is corrupted. The rest of the database will be fine.

RDBMS (MySQL)

At this point, only a single user is supported per DB unless you use the remote server. See the remote server help for more details.

Some familiarity with MYSQL is assumed in the instructions below.

To use RDBMS support with MYSQL:

1.      Install MYSQL and make it accessible on your machine or network. See www.mysql.com for the software (which is free).

2.      Create a database for BORG data in MYSQL. Name it whatever you like.

3.      Create a user with at least select/insert/update/delete access to your database

4.      Create the BORG database tables in your database. This is done by running the SQL contained in the borg.sql file that comes with BORG (you must choose to install the optional RDBMS support). How you run this depends on what tools you are using. From the mysql command line you would do this:

mysql>use your-database-name
mysql>source borg.sql               <=== you need the full path to borg.sql if it is not in your current folder.

5.      BORG looks for a database driver for MYSQL in the CLASSPATH. The driver is ** NOT ** shipped with BORG. You have to download this separately. BORG is tested with MYSQL Connector/J from www.mysql.com/products/connector-j/index.html. To use this, download the latest stable version. Included should be a jar file called something like: mysql-connector-java-3.0.11-stable-bin.jar. Copy it to the lib/ext folder under where the JAVA Runtime is installed. If you don’t correctly do this step, you will get an error saying that the Driver class cannot be found.

6.      Run BORG and set the MySQL database parameters from the database options screen.

7.      You can transfer existing data from BORG local data files by exporting the data to XML, switching  to the new (empty) MySQL DB, and then importing the data from XML.

So, why use RDBMS???? I am not sure. It is much more complicated and requires more administration. I would imagine that some users are comfortable with setting up MYSQL and might want to take advantage of the fact that once the data is in an RDBMS,  ad-hoc data access is easy through SQL. An RDBMS also provides a path forward to allowing multi-user access or even a web interface in the future. As of release 1.5.1, the new remote server supports a multi-user database, and MySQL is recommended as the database in this case.
