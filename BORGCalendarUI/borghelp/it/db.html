Database


Quando avviate BORG per la prima volta, vi verrà chiesto dove salvare il database per immagazzinare i dati di apputamenti, attività e agenda.
Attualmente le scelte disponibili sono:
   - Files locali
   - RDBMS (MySQL)

Ci sono altre scelte sperimentali e non documentate che potete scoprire leggendo il codice. Queste sono disponibili solo da linea di comando, e saranno documentate qui non appena saranno pronte ad un uso pubblico. Un esempio è il database basato sulla memoria che immagazzina i dati solamente in memoria. Questo è utilizzato dall'applet che viene usato per la dimostrazione di BORG, e che può essere visto alla Home Page http://borg-calendar.sourceforge.net.

File Locali


Quasi tutti gli utenti di BORG fanno uso del database predefinito che si basa su File Locali.

Se l'utente sceglie tra le opzioni del DB i File Locali, ed imposta una directory valida BORG crea in questa directory i 3 file seguenti:
   - borg.jdb, per gli appuntamenti del calendario
   - mrdb.jdb, per le attività
   - addr.jdb, per l'agenda

Questi file sono in formato binario. Noi vi consigliamo di fare periodicamente delle copie di BackUp di questi file soprattutto se usate BORG frequentemente, per ovviare a qualsiasi problema del vostro disco, non perchè BORG funzioni male ;-).

I file del DataBase non sono semplicemente degli archivi, ma sono formattati per essere altamente resistenti a possibili corruzioni del disco, problemi di sovrascrittura o insufficienza di spazio, o crash del sistema.
Questi file sono ad accesso casuale. Vari programmi del calendario sovrascriveranno completamente il database nell'eseguire degli aggiornamenti. Questo è soprattutto vero per programmi basati su iCal che usa file locali. Questo non solo è dannoso per le risorse del computer, ma può causare una perdita completa dei dati se qualcosa va storto durante l'aggiornamento.
Quando viene aggiornato il file jdb, quasto avviene solo per il singolo file. Anche se il computer viene spento durante l'aggiornamento, al massimo 1 record sarà corrotto. Il resto del database sarà integro.

RDBMS (MySQL)


Qui ci sono delle istruzioni prese dal file RDBMS.txt incluso con BORG:

Attualmente solo un utente singolo può essere supportato dal DB, anche se un utente può creare DB multipli.
BORG è testato solo con MYSQL, quindi qui ci sono solo istruzione per MYSQL, e presumiamo che abbiate una certa familiarità con questo DBMS.

  1. Installate MYSQL rendendolo accessibile alla vostra macchina o rete. Guardate su www.mysql.com per scaricarlo o per la documentazione (è gratuito).

  2. Create un database in MYSQL per i dati di BORG, chiamandolo come preferite.

  3. Create un utente che abbia almeno accesso al DB per le operazioni: select/insert/update/delete.

  4. Create le tabelle nel DB, utilizzando il file borg.sql presente nel pachetto. Come creare le tabelle dipende dagli strumenti che volete utilizzare Dalla linea di comando dovreste eseguire un comando simile:
      mysql > "nome del DB"
      mysql > "file borg.sql"     <=== è richiesto il percorso completo del file borg.sql, se questo non è nella directory corrente.

  5. BORG cerca un eseguibile per MYSQL nel CLASSPATH. Questo ** NON ** è compreso nel pacchetto di BORG, ma dovete scaricarlo separatamente. BORG &egtrave; testato col MYSQL Connector/J di www.mysql.com/products/connector-j/index.html. Per utilizzarlo, scaricate l'ultima versione stabile che dovrebbe contenere anche un file jar di nome simile al seguente: mysql-connector-java-3.0.11-stable-bin.jar. Copiate questo nella directory lib/ext dove è installata la JAVA Runtime. Se non eseguite correttamente questo passo, otterrete un errore che vi segnala che non viene trovato il Driver class.

  6. Aviate BORG ed impostate i paramentri di MySQL mediante il menu Opzioni.

  7. Potete trasferire i dati contenuti in file mediante l'operazione di Esportazione in file XML che trovate in BORG, scegliete il nuovo DB MYSQL, e poi eseguite l'Importazione da file XML.


Quindi, perchè usare il RDBMS???? Non ne sono sicuro. E' molto più complicato e richiede maggiori operazioni di gestione. Vorrei immaginare che molti utenti trovino comodo usare MYSQL, forse considerando vantaggioso l'uso di un RDBMS ad hoc, e che trovino conveniente utilizzare la sintassi SQL per le loro interrogazioni al DB. Un RDBMS fornisce inoltre la possibilità di una multi-utenza, e non si esclude che in futuro si possa utilizzare per un'interfaccia web.

MYSQL può inoltre essere utilizzato da remoto!! Se avete una rete sicura, non è richiesto alcun lavoro aggiutivo, ma solo inserire il nome del vostro Host nelle impostazioni di BORG. Se volete utilizzarlo in Internet, vi consiglio di farlo usando il tunneling SSH per far transitare in modo sicuro i vostri dati. Il manuale di MySQL descrive come farlo.
